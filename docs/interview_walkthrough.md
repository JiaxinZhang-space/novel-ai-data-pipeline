# 5分钟讲解路线（DEMO）

## 30 秒版本

在这个独立合成 Demo 中，我实现了把合成小说记录和模拟编辑事件转成可训练、可评测、可追踪的数据资产。系统先执行权利字段硬门，再进行文本规范化、精确和近似去重；数据按作者和作品分组切分，避免同作者风格或相邻文本泄漏。之后分别生产 SFT、偏好和隔离评测集，并用 manifest 关联数据、模型占位版本和评测版本。17 项自动化测试验证管道、Schema、去重、集合隔离和公开发布安全。

可以立即声明的是独立 Demo 的设计、代码、测试和合成回归结果；真实授权规模、模型胜率、编辑效率、发布、用户和收入目前暂不能公开。

## 5 分钟版本

### 1. 为什么不能把小说切段后直接训练

- 相邻章节高度相似，随机切分会夸大离线效果。
- 同一作者的语言习惯会让测试集变成“见过的风格”。
- 公开可读不等于允许商业模型训练。
- 只保存最终稿会丢失最有价值的编辑修改理由和失败样本。

### 2. 数据硬门

每条作品必须先匹配来源台账。授权用途、地域、期限、是否允许训练、是否允许衍生内容、是否允许对外发布以及撤回方式均需要明确。权利状态不合格时，管道在任何模型处理之前阻断。

### 3. 去重与隔离

先对规范化文本计算 SHA-256，清理精确重复；再用字符 shingles、MinHash 和 LSH 找近重复候选。切分以作者为最高分组边界、作品为最小不可拆边界，切分完成后才派生场景、SFT 和评测任务。

### 4. 三类数据不是同一份 JSONL

- SFT：instruction、context、response 等监督学习字段。
- 偏好对：同一 prompt 下的 chosen、rejected 及分维度理由。
- 黄金评测：与训练隔离，由编辑或独立读者盲审。

### 5. 如何证明项目有效

先验证数据硬门和泄漏为零，再做基线、单变量消融和盲测。业务层观察编辑修改时间、可发布率和单位可用稿件成本。

## 答疑

### 为什么按作者切分，而不只按作品切分？

同一作者的措辞、结构和人物模板具有明显相关性。只按作品切分仍可能让模型在测试集中遇到训练阶段已经学习过的作者风格，造成评测高估。

### MinHash 能否证明版权安全？

不能。它只是近重复候选发现工具。版权风险还需要来源台账、授权用途审查、撤回机制，以及对生成结果进行更严格的记忆复现检测。

### 一个编辑如何保证标注可靠？

先形成小规模黄金集和清晰评分卡，再保存编辑 diff 与修改原因；高风险或争议样本交给外部盲审。需要报告复审率、仲裁率和不同维度的一致性，不能用一个总分掩盖分歧。

### 如何证明是数据改进，而不是提示词变化？

冻结基础模型、提示模板、采样参数和评测集，只改变一个数据变量；使用相同盲测协议比较，并在 experiment manifest 中记录所有版本和随机种子。

### 删除某位作者的数据后怎么办？

通过权利台账找到作品 ID，再通过数据 manifest 查找所有派生样本和训练分片，并通过模型 manifest 找到消费过该版本数据的模型。隔离相关版本、生成删除报告，并按合同和风险判断重训、回滚或停止使用。
